The Problem

Full Attention Doesn’t Scale

Standard transformers compute attention between every pair of tokens in the context. Double your context length, quadruple the cost. At 64K tokens, that’s over 4 billion attention operations per layer. Previous sparse methods either weren’t trainable (post-hoc approximations) or weren’t hardware-efficient (random scatter/gather patterns that thrash GPU memory). NSA solves both.

The analogy

Full attention is a full table scan on every query — SELECT * FROM context. NSA is more like a well-designed data pipeline: a compressed summary for the big picture, targeted lookups for the important details, and a recent-events buffer for what just happened. Three access patterns, one result.

Core Innovation

Three Branches, One Attention

NSA decomposes attention into three parallel pathways, each handling a different type of context dependency. Their outputs are merged through a learned gating mechanism — the model itself decides how much weight to give each branch for every query.

🗜

Token Compression

Global context → O(n/l)

Groups tokens into blocks of l and compresses each block into a single representation using learned MLP gates. The query attends to these compressed tokens — like reading a summary instead of the full document.

Think columnar aggregation: instead of scanning every row, you read pre-computed block-level statistics.
🎯

Token Selection

Important tokens → O(n · top-k)

Scores every token block by importance (using the compressed representations as a cheap index), then selects the top-k blocks for full fine-grained attention. Only the tokens that actually matter get the expensive computation.

Like a database index scan: use the B-tree to find relevant pages, then fetch only those rows.
📄

Sliding Window

Recent tokens → O(w)

Standard local attention over the most recent w tokens. Captures immediate context — the tokens surrounding the current position. Fixed cost regardless of total sequence length.

A write-ahead log: you always have fast access to the most recent entries.

Learned Gating — How the Branches Combine

O = gcmp · Ocmp + gslc · Oslc + gwin · Owin
The gates g are computed per-query from the input via a linear projection + sigmoid. Each attention head learns its own gating weights — some heads learn to rely heavily on compressed context, others on selection. The model discovers the optimal mix during training.
Why It’s Fast

Hardware-Aligned by Design

Most sparse attention methods look great on paper but are slow in practice. Random token selection means scattered memory reads that destroy GPU throughput. NSA is designed around how GPUs actually move data — every operation maps to coalesced, block-contiguous memory access.

🧱

Block-Contiguous Access

All three branches operate on contiguous blocks of tokens (block size 64). No scattered reads, no random indexing. The compression and selection both use the same block boundaries, so the KV cache is always accessed in GPU-friendly chunks.

ALIGNED TO GPU SRAM
⚡

FlashAttention-Compatible

The selection branch computes blockwise importance scores in SRAM during the forward pass — it piggybacks on the same memory access pattern FlashAttention uses. No extra HBM round-trips for the index lookup.

ZERO EXTRA HBM READS
🔄

Shared Block Indices

The compression branch computes block-level representations that the selection branch reuses as its importance index. Two branches, one data load. The block structure means the same KV cache slice serves both pathways.

COMPUTE REUSE
📊

Custom Triton Kernels

End-to-end fused kernels written in Triton for both forward and backward passes. The selection kernel avoids materializing intermediate attention matrices — it computes scores, selects, and attends in a single kernel launch.

FUSED FORWARD + BACKWARD

NSA’s block-contiguous design keeps the hot path in SRAM. Compressed representations and block indices fit in the on-chip cache; only the selected KV blocks get loaded from HBM. Full attention forces everything through HBM — that’s the bottleneck.

The Selection Kernel — How It Works Under the Hood

A single Triton kernel handles the entire selection branch. No Python-level loops, no materialized intermediate matrices.
Step 1

Load Compressed KVs

Read block-level compressed keys from SRAM (already computed by compression branch).

Step 2

Score Blocks

Compute blockwise importance scores via dot product with query. Runs entirely in SRAM.

Step 3

TopK Select

Pick the top-k highest-scoring blocks. Only these get loaded from HBM.

Step 4

Attend

Run FlashAttention-style computation over selected blocks. Output goes straight to result.

Why the forward pass is fastest

During decoding, the selection branch still needs to score all compressed blocks to find the top-k — that’s the minimum read. During training (backward pass), you need to propagate gradients through the gating and selection, which adds overhead. But the forward pass on a full sequence can batch all queries together and reuse the block index across them. That’s where 11.6× comes from.

So What

Why This Matters

This isn’t a theoretical exercise. DeepSeek published this alongside their production model work. The techniques here are practical enough to ship — and they address the single biggest bottleneck in making LLMs useful: the cost of long context.

💰

Inference Cost Drops

6× faster decoding at 64K context means 6× less GPU time per token generated. For API providers, that translates directly to serving cost. For local deployment, it means running long-context tasks on fewer GPUs.

COST REDUCTION
🔬

Natively Trainable

Unlike post-hoc sparse approximations, NSA trains with the model from scratch. The sparsity pattern is learned, not imposed. This means the model’s sparse patterns are optimal for the actual data distribution, not a researcher’s best guess.

END-TO-END LEARNING
🚀

Long Context Gets Practical

The RULER results show sparse attention doesn’t just match full attention on long context — it beats it. As context windows grow to 128K, 256K, and beyond, this kind of architecture becomes less optional and more necessary.
