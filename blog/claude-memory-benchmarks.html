The Problem

AI assistants forget everything

You explain your project setup, your preferences, the decisions you made last week — and then you do it again tomorrow. Every conversation starts from zero.

Memory systems exist. Mem0, Zep, Letta, and dozens of others promise persistent recall. But most of them extract and compress: they pull out facts, discard the original conversation, and store a summary. The nuance — the debugging session that revealed why you chose one architecture over another — is gone.

What if you just kept everything and built a search engine good enough to find it?

The Solution

What claude-memory does

claude-memory gives Claude Code persistent recall across sessions. It indexes your curated notes and thousands of past conversation archives, so the AI remembers what you discussed last week, last month, or six months ago. Everything runs on your machine — no cloud storage, no API calls for search. One SQLite file.

The search pipeline combines two approaches: keyword matching (FTS5, SQLite’s full-text search engine) catches exact names and dates, while vector similarity (bge-base-en-v1.5 embeddings, compressed 8× via 4-bit quantization) catches paraphrases and related concepts. Results from both are merged using Reciprocal Rank Fusion (RRF) — a technique that combines ranked lists without needing to calibrate their scores.

I built it because I use Claude Code every day and got tired of re-explaining context. Then I got curious: how does it compare to the systems everyone’s tweeting about? Mem0, Zep, Letta, MemPalace — they all publish benchmark numbers. So I ran the same benchmarks.

Reading Guide

How to read these results

R@5 / R@10 — Recall at 5 or 10: did the correct answer appear in the top 5 or 10 search results? Measures the retrieval engine alone, independent of the LLM.
J-score (LLM-as-Judge) — A judge LLM reads the question, gold answer, and generated answer, then scores CORRECT or WRONG.
F1 — Token-level overlap between generated and gold answer. Strict — even correct answers score low if worded differently.
E2E (End-to-End) — The full pipeline: retrieve context, generate an answer, judge it. Tests retrieval + reasoning together.

The Benchmarks

Two benchmarks, two different tests

LongMemEval

ICLR 2025. 500 questions across 6 types. Tests single-session recall, multi-session synthesis, temporal reasoning, knowledge updates. The standard benchmark for AI memory.

Think: "What degree did I graduate with?" buried in 50 sessions of conversation.

LoCoMo

ACL 2024, Snap Research. 10 conversations, 1,986 questions, 5 categories. Tests multi-hop reasoning, temporal inference, adversarial questions.

Think: "What would Caroline likely pursue in education?" requiring synthesis across months of casual conversation.

LongMemEval tests can you find the needle. LoCoMo tests can you reason across scattered needles. Together they cover the full spectrum of what a memory system needs to do.

Head-to-head with MemPalace

MemPalace is the closest comparable system — they also store raw verbatim text and search it with embeddings. But they use bge-large (335M params, full precision) and ChromaDB. We use bge-base (110M params, 4-bit quantized) and SQLite. A 3× smaller model with lossy quantization.

Cross-encoders hurt

They’re trained on the wrong domain. Models like mxbai-rerank and MiniLM-L-6 learned relevance from MS MARCO — web search queries paired with web pages. Conversational memory looks nothing like web search results.

The cross-encoder confidently promotes chunks that look relevant by web-search standards and demotes chunks our RRF correctly ranked. R@5 improves slightly, but R@10 tanks.

Haiku reranking works

An LLM actually reads. Haiku understands that “I went to the LGBTQ group on May 7th” answers “When did Caroline go to the support group?” A cross-encoder trained on web search can’t make that connection.

But adding Haiku reranking means API calls, cost, and latency — making our system a copy of MemPalace’s approach rather than a genuinely different architecture.

Our edge is the hybrid search architecture, not stacking more models on top. FTS5 keyword matching catches what vector search misses. RRF fusion combines both signals without needing a reranker to fix mistakes. There’s nothing to fix. The architecture is the advantage.

Why It Works

The uncomfortable reason simple beats complex

Every competitive memory system uses an LLM to manage memory. Mem0 extracts facts. Zep builds knowledge graphs. Letta summarizes into a filesystem. They all start from the assumption that you need AI to decide what to remember.

The extraction step is where information dies. When Mem0 extracts “user prefers PostgreSQL” and throws away the conversation about why — the alternatives considered, the tradeoffs discussed, the team constraints mentioned — that context is gone forever. When a question asks about database tradeoffs, there’s nothing to retrieve.

My system doesn’t extract anything. It keeps the raw conversations and searches them directly. Nothing is lost. The hybrid search finds what you need, and the LLM at the end has the full context to reason with.

20pp
The gap between claude-memory (87.3%) and Mem0 (66.9%) on LoCoMo — the benchmark Mem0 chose to publish against. The entire gap is information that Mem0’s extraction step threw away.

This only works because my use case is personal: one user, thousands of conversations. At Mem0’s scale — millions of users, millions of conversations — you can’t keep everything and search it. Extraction and summarization become necessary. But the threshold is much higher than anyone realized. For personal AI memory, raw text search is the right architecture.

Honest Limitations

What I’m not claiming

This scales

It doesn’t. Brute-force cosine similarity on all embeddings in RAM works for ~115K chunks. It won’t work for a million. This is a personal tool, not a platform.

The retrieval is novel

FTS5 + vector + RRF is a well-known combination. bge-base is off-the-shelf. TurboQuant is someone else’s paper. The individual pieces aren’t new. The execution is what works.

The LLM doesn’t matter

It does. With Gemma 4 31B, the same retrieval scores 76.2% on LongMemEval instead of 100%. The retrieval engine is ours; the generation quality belongs to Sonnet.

These results are independently verified

Not yet. The code is open source and the results are saved to JSONL, but no third party has reproduced them. Single run, not mean ± stdev.

The Takeaway

What I actually learned

I spent a weekend reading the MemPalace README, running their benchmarks on my system, testing cross-encoder reranking, tuning parameters, and documenting everything honestly. Here’s what I learned:

1. Don’t throw away your conversations. Every memory system that extracts, summarizes, or compresses loses information. For personal use, keep the raw text and search it.

2. Hybrid search is underrated. Vector-only search misses exact matches (names, dates, specific terms). Keyword-only search misses semantic similarity. Combining them with RRF beats both — and it beat MemPalace on R@10 despite using a 3× smaller model.

3. More models ≠ better results. Cross-encoder reranking, the standard technique for improving retrieval, made our R@10 worse by 3.6pp. Sometimes the pipeline you have is already the right one.

4. Benchmark honestly or don’t benchmark. MemPalace got caught with fake token counts and unreproducible claims. The memory systems space is full of inflated numbers. Run the same benchmarks, report the same metrics, publish the scripts.

For personal AI memory — one user, thousands of conversations — raw text search beats every extraction-based system on standard benchmarks. The funded startups over-engineered for scale they don’t need yet, and lost information in the process.

The system is open source. The benchmarks are reproducible. The results files are committed. LongMemEval_S is likely saturated at 100%, but temporal reasoning (68.8% on LoCoMo) remains the hardest category, and fully-local inference with smaller models is the next frontier.
