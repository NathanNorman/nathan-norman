Teaching a Network to Play Wingspan

Nine training runs, three months, and one broken ruler. An AlphaZero implementation for a complex card game: what broke, what worked, and why the numbers we watched for most of it were wrong.

🦅 446 birds 📊 9 training runs 🔬 MCTS + AlphaZero 📅 May 2026

The Game and the Goal

Wingspan is a tableau-building card game where players draft bird cards, place them across three habitats, and activate chain reactions of special powers. Each bird has a food cost, an egg capacity, a habitat preference, a power type, and sometimes a chain of effects that triggers other birds. After four rounds and ~190 decisions, whoever has built the most efficient scoring engine wins.

The benchmark is SmartHeuristicAgent: a hand-coded agent that uses prospective scoring, food accumulation strategy, round-awareness, and habitat preference. It wins 90.5% of games against a random opponent and averages a score of 55. Beating it consistently requires planning several turns ahead, the kind of look-ahead that 50 tree search simulations cannot provide.

Phase 1: Supervised Pretraining

Before any self-play, the network spends 200,000 gradient steps imitating SmartHeuristicAgent across 900,000 generated games. The policy head learns to copy the heuristic's move choices; the value head learns to predict game outcomes from position. After 17 hours on a c7i.8xlarge, policyLoss drops from 22.6 to 0.245 and valueLoss reaches 0.007.

Then we test it: run HeuristicEval using greedy argmax against SmartHeuristicAgent. The win rate is 14.4%. Against a random agent, it's 50%. Exactly coin-flip.. The policy head has memorized a move distribution, but argmax of that distribution plays at random quality. MCTS is not optional. The policy head is a prior for tree search, not a standalone decision-maker.

Building the MCTS Loop

The AlphaZero training loop is a feedback cycle: the network generates self-play games using MCTS, those games produce training targets, and gradient descent updates the network. Repeat until the network gets good enough that its own games are better training data than yesterday's.

The AlphaZero training loop. MCTS uses the network's policy prior and value estimate to build a search tree; the resulting visit distribution becomes the policy training target.

The Network Architecture

The network is a flat MLP with a shared trunk and 20+ per-decision-type policy heads, plus one value head:

Input (~400-dim observation)
  └─ trunk: Linear(→512) → ReLU → Linear(→512) → ReLU → Linear(→256) → ReLU
       ├─ policy_PLAY_BIRD:      Linear(256→70)   → visit distribution
       ├─ policy_CHOOSE_FOOD:    Linear(256→6)    → visit distribution
       ├─ policy_POWER_CHOICE:   Linear(256→100)  → visit distribution
       ├─ … (20+ heads, one per DecisionType)
       └─ value:                 Linear(256→1)    → sigmoid → [0,1]

One detail that causes problems: the value head and all 20+ policy heads compete for the same 256-dimensional trunk output. This matters a lot (more on it in the bugs section).

How MCTS uses the network at each decision

For each game decision, MCTS runs N simulations. Each simulation walks the tree using PUCT selection (which balances exploitation of high-value nodes with exploration of under-visited ones), eventually reaching a leaf node. At the leaf, the network evaluates the board state: the policy head provides prior probabilities for each legal action, and the value head estimates the probability the current player wins from here. The value is backed up through the tree, updating visit counts and Q-values. After N simulations, the visit distribution at the root becomes the training target π(a) for that decision.

Dirichlet noise is added to root priors during training to ensure exploration. The noise alpha scales with the action space: α = 10 / |actions|, so a 5-action decision gets α=2.0 (broad noise) while a 50-action decision gets α=0.2 (concentrated noise).

Three bugs that took months to find

Bug: HeuristicEval at 0 sims

The evaluator was hardcoded to use mctsSimulations=0, measuring greedy argmax of the MCTS prior. A network trained to spread probability mass across actions always scored 0% wins. The eval was measuring the wrong thing entirely.

Fix: Pass config sims to evaluator

Always evaluate with the same MCTS wrapper used during training. The policy head is a prior, not a policy. Greedy argmax is meaningless without the search that gives it structure.

Bug: c_puct ≈ 0.07 effective

Using the DeepMind log-scale formula with C_BASE=19652 at 50 simulations: cPuct_eff = 1.0 × √(2 ln(19703/19652)) = 0.071. Near-greedy tree search. MCTS barely improved on the raw prior.

Fix: c_puct=5.0 (flat)

The log-scale formula is calibrated for 800 simulations. At 50 sims, use a flat coefficient. c_puct=5.0 gives effective exploration comparable to AlphaZero's configuration at full sim count.

The Plateau, or So We Thought

After fixing the evaluation bug and the c_puct formula, training started producing real results. The network won games. HeuristicEval (net vs SmartHeuristic, 100 games, 50 MCTS simulations) showed net avg scores rising from the low 20s toward 24-25.

Then it stopped. Every run, every configuration, the same result: net avg peaked around 24-25 somewhere between iter 20 and iter 80, then declined or stagnated. Six runs across two months, all hitting the same ceiling. We changed the opponent mix, the replay buffer size, the historical opponent fraction, the instance type. The ceiling didn't move.

The Measurement Problem

Every HeuristicEval during training used 50 MCTS simulations, the same count used during training. The logic seemed sound: test the same way you train. But MCTS is a search algorithm. The strength of the resulting play scales with simulation count. Running 50 sims at eval time wasn't measuring how good the network was. It was measuring how good the network was given a very shallow search.

What We Actually Broke Along the Way

Each of these bugs was silent: the training loop kept running, losses kept moving, and the problem only showed up in eval results or crash logs. None of them were obvious from the code.

DNNL_DEFAULT_FPMATH_MODE=BF16
Added as an inference optimization for the ARM instance. The environment variable applies to all oneDNN primitives, including backward GEMMs. BF16 has 7 mantissa bits; small value head gradients underflowed to zero. The value head froze at valueLoss=0.60 for 125 iterations while policyLoss kept declining.
Fix: Remove entirely. BF16 mode is documented for inference, never for training. The DJL page that recommends it is titled "Inference Performance Optimization."
Reading freed NDArrays after gc.backward()
Loss scalar values were extracted after the gradient collector's use{} block closed. DJL frees NDArrays when the collector exits. Without BF16, freed memory actually clears, returning NaN. BF16 mode had left the memory readable, masking the bug entirely. First run without BF16: both losses reported NaN, weights corrupted on step 1.
Fix: Extract scalar values inside the use{} block, before gc.backward().
80:1 gradient ratio (value head starvation)
Setting value loss weight to 0.25 with 20 decision types produces an 80:1 effective gradient ratio on the trunk (20 policy heads × 4x weight). The trunk specializes for policy-useful features. The value head, reading a policy-specialized 256-dim vector with no private hidden layer, sees an increasingly hostile input distribution. valueLoss rises over 100+ iterations.
Fix: Either raise value weight back toward 1.0, or add a private hidden layer to the value head so it can transform policy-specialized trunk features.
Heuristic opponents poisoning avgZ
With 50% of training games using SmartHeuristicAgent as opponent, those games had z≈0.05 (the net lost badly). The replay buffer filled with "I always lose" examples. The value head learned to output near-zero everywhere. MCTS, relying on value estimates for leaf evaluation, became effectively greedy. HeuristicEval dropped to 16-18 net avg, worse than the starting checkpoint.
Fix: Zero heuristic opponents. Use only self-play, historical pool, and random agents. avgZ immediately stabilized at 0.48-0.50.
Heuristic anchor games in replay buffer
Running 50 heuristic-vs-heuristic games per iteration to prevent catastrophic forgetting sounded sensible. These produce 1-hot policy targets (deterministic heuristic choice). MCTS games produce visit distribution targets (soft probability). Both landed in the same replay buffer. The policy head received contradictory gradient signals for 40 iterations, policyLoss stalled at 14+.
Fix: Zero anchor games. The supervised pretrained checkpoint already provides the base knowledge. MCTS should build from there without re-injecting 1-hot supervision.
tau=1.0 training exploratory agents
Temperature τ=1.0 means moves are sampled proportionally to visit counts. The network never commits to its best move during training. With 1.9M training positions all labeled with exploratory distributions, the network learned "spread probability mass like a 50-sim explorer." Evaluated near-greedily against SmartHeuristic, it played stochastically against a deterministic opponent. policyLoss of 13.5-14 nats ≈ near-uniform distributions (ln(1M) ≈ 13.8).
Fix: Let tau decay run past iter 51. v8 was the first run to survive long enough to see tau drop below 0.5. Net avg rose from 25.4 → 27.7 directly in step with the decay.

v9: The First Clean Run

With the measurement problem diagnosed, the fix for training was clear: use 200 simulations during training itself, not just at eval time. More sims during self-play means deeper trees, sharper visit distributions as policy targets, and less noise in the learning signal. The policy head trains toward "what MCTS recommends after real search" rather than "what a 50-sim random walk happens to visit."

v9 starts from v8's best checkpoint (38.5 net avg at 200-sim eval) and runs 200 sims per move during generation. Gen time rises from 120s to ~480s per iteration, 4× slower, but the training signal quality is meaningfully higher.

Why self-play scores rose so much at 200 sims

Self-play avg scores jumped from 28-32 (v8) to 35-38 (v9). Both players improved.. With deeper search, both sides find better moves, and the resulting games are higher-quality. This feeds back into better training targets: visit distributions from 200-sim trees are sharper and more informative than from 50-sim trees. The policy head trains against more clearly differentiated action preferences, so it learns more per gradient step.

At 50 sims with branching factor B=20, the top move gets ~15-20 visits and most get 0-1. The visit distribution is mostly prior-shaped with small MCTS corrections. At 200 sims, the top move gets 60-80 visits, creating a meaningfully peaked distribution that the policy head can actually learn from.

Four Things That Transfer

1. Eval sim count is not training sim count

MCTS strength scales with simulation count. Evaluating at 50 sims when training at 50 sims seems consistent, but you're measuring search quality at the cheapest setting, not network quality. Run eval at 2-4× training sims to see what the network actually knows. The 10-point gap between 50-sim and 200-sim eval was the most consequential finding in this entire project.

2. tau=1.0 trains exploratory agents

AlphaGo Zero's temperature schedule is per-game (τ=1.0 for first 30 moves, then greedy). Holding τ=1.0 for 50+ full training iterations trains the network to sample proportionally to visit counts, not to select best moves. The resulting agent plays stochastically against deterministic opponents. Tau decay needs to happen. The effect is measurable and rapid once it does.

3. BF16 is an inference optimization, full stop

DNNL_DEFAULT_FPMATH_MODE=BF16 appears in DJL's inference tuning documentation for a reason. It routes all oneDNN GEMMs through BF16, including weight gradient computation. For batch-1 inference on ARM, it's a meaningful speedup. For training, it corrupts the backward pass for any head whose gradients are small-magnitude, exactly the value head, which always gets smaller gradients than the combined policy signal.

4. A single Linear value head cannot co-exist with 20+ policy heads

Every serious AlphaZero implementation uses a hidden layer in the value head. The reason is structural: the shared trunk gets pulled toward policy-useful representations by whichever gradient signal is largest. A value head with no private transformation capacity reads whatever the policy gradient leaves behind. At 0.25 value weight with 20 decision types, that's an 80:1 gradient ratio. The trunk moves faster than a single linear layer can track. The hidden layer gives the value head its own feature space to work with, independent of trunk specialization.
